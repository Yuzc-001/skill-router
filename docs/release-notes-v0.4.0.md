# Skill Router v0.4.0 Release Notes

日期：2026-03-21  
版本：`v0.4.0`

---

## Summary

`v0.4.0` is the release where Skill Router stops feeling like a taxonomy exercise and starts feeling like a product.

This release is not about routing more often.
It is about routing at the right moments, with the right amount of presence, and with clearer product boundaries.

In short:

# Skill Router v0.4.0 is about reducing selection waste without replacing judgment.

---

## What changed in v0.4.0

### 1. Product positioning was tightened
Skill Router is now explicitly framed as a:
- low-presence routing layer
- conflict-aware decision layer
- judgment-preserving support skill

It is no longer presented as a system that should classify everything or sit in front of every task.

### 2. The boundary got sharper
The release makes one rule explicit:

# When skill choice is not the problem, Skill Router should disappear.

This matters because a routing layer becomes harmful when it inserts itself into simple work that should just proceed.

### 3. Conflict-set thinking replaced taxonomy sprawl
`v0.4.0` shifts the center of gravity from “more categories” to “real recurring overlap”.

Skill Router is now clearest when resolving:
- browser-control overlap
- search overlap
- memory / note / SOP overlap
- methodology / process overlap

The product now treats routing detail as valuable only when it changes the default decision.

### 4. Installed reality became the public default
The release strengthens the product rule:

# Installed reality beats local preference.

Skill Router now clearly prefers:
- what is already installed
- what is already known to work in this environment
- what is already sufficient

before discovery is even considered.

### 5. Discovery was demoted to its proper role
Discovery is now explicitly framed as a fallback for insufficiency, not a novelty engine.

That means:
- no discovery just because something newer might exist
- no discovery just because comparing options feels clever
- discovery only when installed capabilities are genuinely not enough

### 6. Public-facing docs now tell one coherent story
The README, `SKILL.md`, and core docs now align around the same product shape:
- low-presence
- reuse-first
- conflict-aware
- quiet-by-default
- judgment-preserving

This makes `v0.4.0` feel less like a pile of routing rules and more like a coherent work.

---

## What v0.4.0 claims

Skill Router v0.4.0 claims that:
- not every task deserves routing
- installed skills should be preferred before discovery
- routing should focus on real selection conflicts
- discovery should only happen for real insufficiency
- unfamiliar candidates should be vetted before recommendation
- the routing layer should reduce waste without replacing agent judgment

---

## What v0.4.0 does not claim

Skill Router v0.4.0 does **not** claim:
- universal skill taxonomy completeness
- maximal classification of every possible task
- automatic superiority over direct agent judgment
- constant intervention in normal execution flow

The product is better described as:

# a quiet routing calibration layer

not:

# a universal meta-governor for all work

---

## Release significance

This release matters because it changes the shape of the product.

Before, Skill Router could be read as a useful routing document.
Now it reads much more like a finished product stance:
- clearer boundary
- clearer role
- clearer disappearance conditions
- clearer relationship to judgment

That shift is what makes `v0.4.0` feel like a real work.
