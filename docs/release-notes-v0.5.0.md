# Skill Router v0.5.0 Release Notes

日期：2026-03-22  
版本：`v0.5.0`

---

## Summary

`v0.5.0` is the release where Skill Router stops trying to justify itself through routing doctrine and starts justifying itself through waste reduction.

This release is not about routing more elegantly.
It is about speaking less, deciding faster, and only intervening when the routing decision changes the next real action.

In short:

# Skill Router v0.5.0 is about reducing unnecessary skill-selection cost.

---

## What changed in v0.5.0

### 1. The product center was reset
Skill Router is no longer centered on:
- taxonomy expansion
- routing philosophy for its own sake
- long explanation of how to choose

It is now centered on one runtime purpose:

# reducing unnecessary decision cost in crowded skill environments

### 2. The skill contract was rewritten
`SKILL.md` was fully rewritten around a much smaller runtime path:
1. decide whether routing is needed at all
2. prefer the strongest installed default
3. test genuine insufficiency
4. discover only for a real gap
5. vet unfamiliar candidates before recommendation

This replaces the older, more doctrine-heavy routing model.

### 3. Silence became a core product behavior
`v0.5.0` treats silence as a feature.

The main public rule is now:

# if routing will not change the next real action, Skill Router should not speak

This is one of the most important changes in the release.

### 4. The task-to-skill map was demoted
The routing map is no longer presented as a growing universal routing table.
It now exists only as a lightweight local-default record for recurring overlap clusters that genuinely reduce future routing waste.

### 5. High-value conflicts were narrowed
The main path now focuses only on conflict types that materially matter:
- installed vs discover
- dedicated vs general
- platform-specific vs generic
- recurring overlap cluster
- unfamiliar install vs safe reuse

If a distinction does not change the default next action, it is out of scope for the main path.

### 6. The docs were rebuilt around runtime value
The main documentation set was rewritten to align with the reset:
- `docs/skill-router-v0.5-ruthless-reset.md`
- `docs/why-skill-router.md`
- `docs/how-it-works.md`
- `docs/public-surface.md`
- `docs/use-cases.md`
- `docs/with-vs-without.md`
- `references/resolution-order.md`
- `references/task-to-skill-map.md`
- `references/micro-routing-examples.md`
- `references/local-overrides-example.md`

The repo now tells a much more coherent story about why this product should exist.

---

## What v0.5.0 claims

Skill Router v0.5.0 claims that it can:
- reduce repeated discovery of already-sufficient installed skills
- reduce repeated comparison in recurring overlap clusters
- reduce avoidable installation churn
- reduce routing commentary when no real decision change is happening
- keep discovery constrained to real capability gaps
- insert a vetting gate before unfamiliar candidates are recommended

---

## What v0.5.0 does not claim

Skill Router v0.5.0 does **not** claim:
- universal taxonomy completeness
- maximal routing coverage
- intervention on every task
- superiority over direct execution when the path is already obvious
- value through explanation alone

The product is better described as:

# a low-presence default resolver for crowded skill environments

not:

# a universal meta-routing doctrine

---

## Release significance

This release matters because it changes the basis on which the product is allowed to exist.

Before, Skill Router could still be read as a sophisticated routing framework.
Now, the standard is much harsher:

# if it does not reduce real waste, it does not deserve to speak

That makes `v0.5.0` a much more opinionated and much more serious product reset.
